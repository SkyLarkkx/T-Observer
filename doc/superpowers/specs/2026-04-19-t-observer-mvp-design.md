# 教师听课评课记录与分析系统 MVP 设计说明

> 课题名称：基于 Spring Boot 和 Vue 3 的教师听课评课记录与分析系统的设计与实现
> 文档日期：2026-04-19
> 适用阶段：第一阶段 MVP 实现

## 1. 设计目标

本设计文档用于定义“教师听课评课记录与分析系统”第一阶段 MVP 的实现边界、模块划分、核心流程和技术方案。MVP 聚焦毕业设计题目的核心闭环，优先完成“任务发布、听课填报、审核流转、结果汇总、雷达图生成”。

本阶段目标不是一次性覆盖全部需求，而是在现有前后端脚手架基础上实现一个可演示、可验收、可继续扩展的首版系统。

## 2. MVP 范围

### 2.1 本阶段纳入范围

1. 简化登录与角色鉴权
2. 听课任务创建、分配与列表查看
3. 听课记录草稿保存与提交
4. 组长审核通过与退回
5. 优点项与待改进项汇总
6. 教师教学能力雷达图生成

### 2.2 本阶段暂不纳入范围

1. 学校统一身份认证接入
2. 被听课教师独立门户
3. 申诉与复核流程
4. 消息提醒与逾期预警
5. PDF 导出
6. 复杂统计报表
7. 自动归档与超时补录流程

## 3. 总体设计思路

系统采用前后端分离架构。

- 前端使用 Vue 3、TypeScript、Vite 构建业务页面
- 后端使用 Spring Boot + MyBatis 提供 RESTful API
- MySQL 负责核心业务数据持久化
- Redis 在 MVP 中主要作为登录态或缓存预留，不强依赖复杂缓存设计

首版以规范实现为主，优先确保业务闭环完整、结构清晰、便于答辩展示，而不是追求一次性做全所有扩展功能。

## 4. 角色设计

MVP 使用本地账号密码登录，并采用固定角色权限控制。

### 4.1 教研组组长

具备以下权限：

1. 创建和查看听课任务
2. 分配听课人
3. 审核听课记录
4. 查看教师分析结果和雷达图

### 4.2 教研组成员

具备以下权限：

1. 查看分配给自己的听课任务
2. 填写听课记录
3. 保存草稿
4. 提交记录
5. 查看自己的记录状态

### 4.3 管理员

MVP 中仅保留系统演示所需的基础账号管理定位，不单独展开复杂后台页面。

## 5. 核心业务模块

### 5.1 登录模块

系统采用简化登录方案，用户使用系统预置账号密码登录。登录成功后，后端返回用户基本信息和角色信息，前端根据角色加载对应菜单和页面。

### 5.2 听课任务模块

组长创建任务时填写任务标题、听课人、被听课教师、课程名称、听课时间、截止时间和备注信息。任务创建后，成员只能查看分配给自己的任务，组长可查看任务整体进度。

### 5.3 听课记录模块

成员基于任务填写听课记录。每条记录包含 5 个固定维度评分和 3 个文本区域：

1. 优点项
2. 待改进项
3. 改进建议

记录支持保存草稿与正式提交两种动作。

- 保存草稿时允许内容不完整
- 提交时必须通过完整性校验和业务规则校验

### 5.4 审核模块

组长对已提交记录执行审核。审核结果分为：

1. 通过
2. 退回

退回必须填写明确原因。审核通过后的记录才能进入分析样本。

### 5.5 分析模块

系统按“教师 + 周期”聚合已通过记录，输出：

1. 优点汇总
2. 待改进项汇总
3. 5 个维度平均分
4. 教学能力雷达图
5. 简要分析结论

如果样本数量不足 3 条，则不生成雷达图，仅展示“样本不足”的提示。

## 6. 固定评课维度设计

MVP 采用固定 5 维度方案，不做动态维度配置，以降低实现复杂度并保证答辩演示稳定性。

固定维度如下：

1. 教学设计
2. 课堂组织
3. 教学内容
4. 互动反馈
5. 教学效果

评分规则如下：

1. 分值范围为 1.0 到 5.0
2. 步长为 0.5
3. 提交时 5 个维度必须全部填写

## 7. 页面结构设计

MVP 前端页面控制在 6 个核心页面。

### 7.1 登录页

用户输入账号密码登录，登录成功后跳转到角色首页。

### 7.2 成员任务列表页

成员查看分配给自己的任务，并按状态筛选：

1. 待填写
2. 草稿
3. 已提交
4. 已退回
5. 已通过

### 7.3 听课记录填写页

页面上方展示任务基础信息，下方展示评分区域和文本区域，底部提供“保存草稿”和“提交”按钮。

### 7.4 组长任务管理页

组长查看任务列表、创建任务、筛选状态，并查看任务完成进度。

### 7.5 审核页

组长查看听课记录详情，执行通过或退回操作。退回时必须填写退回原因。

### 7.6 分析页

组长按教师和周期查询分析结果。页面展示优点汇总、待改进项汇总、维度均分和雷达图；若样本不足，则展示文字提示。

## 8. 核心流程设计

### 8.1 成员流程

1. 登录系统
2. 查看分配给自己的任务
3. 填写听课记录
4. 保存草稿或提交
5. 若被退回，则修改后重新提交

### 8.2 组长流程

1. 登录系统
2. 创建听课任务
3. 查看成员提交情况
4. 审核记录
5. 查看指定教师的分析结果

## 10. 状态流转设计

### 10.1 任务状态

建议使用以下任务状态：

1. `PENDING`
2. `IN_PROGRESS`
3. `COMPLETED`

状态更新原则：

- 任务创建后为 `PENDING`
- 成员开始填报后为 `IN_PROGRESS`
- 关联记录审核通过后可视为 `COMPLETED`

### 10.2 记录状态

MVP 建议使用以下记录状态：

1. `DRAFT`
2. `SUBMITTED`
3. `RETURNED`
4. `APPROVED`

状态约束：

- `DRAFT` 和 `RETURNED` 允许编辑
- `SUBMITTED` 不允许提交人继续编辑
- `APPROVED` 只允许查看，不允许修改

## 11. 业务规则落地设计

MVP 需要严格落实以下规则：

1. 提交记录时必须填写 5 个维度评分、优点项、待改进项和改进建议
2. 评分必须在 1.0 到 5.0 之间，步长为 0.5
3. 同一听课人对同一任务只能存在 1 条有效记录
4. 退回操作必须填写原因
5. 分析时只统计 `APPROVED` 状态记录
6. 样本数小于 3 条时不生成雷达图数据
7. 关键状态变更必须写入审计日志

## 13. 前后端实现结构

### 13.1 后端分层

建议采用以下目录结构：

- `controller`：接口控制层
- `service`：业务逻辑层
- `mapper`：MyBatis 数据访问层
- `entity`：数据库实体对象
- `dto`：接口入参对象
- `vo`：接口出参对象
- `common`：统一响应、异常、工具类

### 13.2 前端分层

建议采用以下目录结构：

- `views`：业务页面
- `components`：可复用组件
- `api`：接口请求封装
- `stores`：登录态与用户状态
- `router`：路由配置
- `types`：类型声明

## 14. 异常处理设计

系统异常处理建议分为 3 层。

### 14.1 前端表单校验

前端负责校验必填项、分值范围和文本长度，减少无效请求。

### 14.2 后端业务校验

后端负责校验：

1. 是否重复提交
2. 当前状态是否允许执行目标操作
3. 退回原因是否为空
4. 样本数量是否满足雷达图生成条件

### 14.3 统一返回封装

后端统一返回 `code`、`message`、`data` 三段式结构，前端统一展示操作结果提示。

## 15. 测试与演示策略

### 15.1 后端测试重点

重点覆盖以下场景：

1. 登录成功与失败
2. 草稿保存
3. 提交时字段校验失败
4. 退回时未填写原因
5. 仅 `APPROVED` 记录进入分析
6. 样本不足时不生成雷达图

### 15.2 前端验证重点

重点验证以下流程：

1. 登录后按角色跳转
2. 任务列表筛选
3. 听课记录保存草稿与提交
4. 审核通过与退回
5. 分析页展示雷达图或样本不足提示

### 15.3 演示数据准备

为确保毕业设计演示顺畅，系统应准备初始化演示数据，包括：

1. 多个角色账号
2. 多条听课任务
3. 多条听课记录
4. 可用于生成雷达图的已通过样本数据

## 16. 预期实现结果

完成 MVP 后，系统应能够稳定支持以下业务闭环：

1. 组长创建任务
2. 成员填写并提交听课记录
3. 组长审核记录
4. 系统按教师汇总结果
5. 系统生成教学能力雷达图

该实现结果能够直接支撑课题题目中的三项原始需求：

1. 教研组成员填写听课表
2. 系统汇总优点和待改进项
3. 系统生成教师教学能力雷达图
